# Phase 30: Аналитика — дашборд «Топ-30 SKU в нише» — Specification

**Created:** 2026-07-13
**Ambiguity score:** 0.167 (gate: ≤ 0.20)
**Requirements:** 12 locked

## Goal

Аналитик загружает 6 файлов отчёта WB «Сравнение карточек» + указывает MPSTATS-токен, сервис по 30 конкурентным SKU собирает воронку продаж, выручку, выкуп, медианную цену, листинг-фото, характеристики и позиции в поиске (органика/реклама), сохраняет прогон и показывает полноэкранный дашборд из 5 вкладок с единой сортировкой и графиками динамики; результат можно выгрузить в PDF. Раздела `/analytics` и `ERP_SECTION.ANALYTICS` сегодня не существует — они создаются в этой фазе.

## Background

Текущее состояние кода (scout 2026-07-13):
- **Раздела аналитики нет.** `ERP_SECTION` (prisma/schema.prisma) содержит 15 значений (PRODUCTS…FINANCE), `ANALYTICS` отсутствует. Разделы регистрируются в `lib/sections.ts` (префикс URL → секция, edge-safe для middleware) + `lib/rbac.ts` + навигация.
- **Скан карточек WB уже реализован:** `lib/wb-api.ts` дёргает `card.wb.ru/cards/v4/detail` через **curl** (Node `fetch` блокируется TLS-fingerprint), отдаёт цену/СПП/рейтинг/отзывы per nmID; фото WB раздаются basket-CDN по nmID. ТЗ §2.3 требует переиспользовать этот механизм, а не писать новый.
- **MPSTATS в проекте нет** — новая внешняя интеграция (токен `X-Mpstats-TOKEN`).
- **Графики:** `recharts` уже используется в 8 местах (+ shadcn `components/ui/chart.tsx`).
- **PDF:** зависимость `pdfkit` уже в `package.json`.
- **Загрузка/парсинг файлов:** паттерн есть (`lib/parse-ivanovo-excel.ts`, `lib/bank-import/`, `lib/wb-commission-iu-parser.ts`); detail-файлы — JSON, парсятся `JSON.parse` в server action.
- Функционально раздел **независим от товаров компании** (`Product`/`MarketplaceArticle`) — анализируются произвольные конкурентные nmID.

Первичный источник требований: `.planning/phases/30-analytics/30-TZ-source.md` (ТЗ v3).

## Requirements

1. **Загрузка отчёта «Сравнение карточек»**: пользователь загружает 6 JSON-файлов detail; их состав полностью определяет топ-30 (6×5=30 SKU), без доп. отбора.
   - Current: механизма загрузки/парсинга detail-JSON нет
   - Target: форма загрузки 6 файлов → валидация структуры (`data.salesFunnel.byDay/byWeek/byMonth`, `data.commonParams`, `data.searchQueries`) → извлечение 30 уникальных nmID + посуточной воронки + commonParams
   - Acceptance: загрузка 6 валидных файлов даёт ровно 30 SKU; файл с неверной структурой/дубликатом nmID отклоняется с явным сообщением; при <6 файлах или <30 уникальных SKU прогон не стартует

2. **Движок метрик воронки (правила расчёта ТЗ §3)**: средние и производные метрики считаются по фиксированным правилам.
   - Current: расчётного движка нет
   - Target: pure-функции в `lib/analytics/`: объёмы (показы/заказы/сумма) = месяц ÷ 30; конверсии и выкуп = «от сумм» (CTR=Σпереходов/Σпоказов, клик→корзина=Σкорзин/Σпереходов, корзина→заказ=Σзаказов/Σкорзин, клик→заказ=Σзаказов/Σпереходов, выкуп=Σвыкупов/Σзаказов); конверсия клик→заказ также = клик→корзина × корзина→заказ; медианная цена = средняя за период с поправкой −3% (WB-Кошелёк)
   - Acceptance: golden-тесты на фикстуре с известными суммами дают ожидаемые значения; конверсия клик→заказ «от сумм» совпадает (в пределах округления) с произведением двух конверсий; цена = report_price × 0.97

3. **Интеграция MPSTATS — позиции в поиске**: по каждому из 30 SKU тянутся позиции по запросам (органические и рекламные раздельно), окно дат = период файлов.
   - Current: MPSTATS-клиента нет
   - Target: клиент MPSTATS (заголовок `X-Mpstats-TOKEN`), 1 SKU за запрос (+пагинация), окно дат = период `byDay` из файлов; список запросов = объединение запросов, по которым ранжируются 30 SKU, с частотностью > 500; для каждого запроса — дневная органическая позиция и рекламная позиция с параметрами (CPM, тип размещения, позиция буста)
   - Acceptance: для тестового SKU клиент возвращает дневной ряд позиций, разделённый на organic/ad; запросы с частотностью ≤ 500 отфильтрованы; счётчик обращений к API отражает ≥30 запросов; лимит-ошибка (429/лимит тарифа) обрабатывается без падения

4. **Скан карточек конкурентов (реюз механизма)**: 5 фото листинга + характеристики по каждому nmID.
   - Current: `lib/wb-api.ts` умеет тянуть цену/рейтинг per nmID через curl card.wb.ru; листинг-фото и характеристики для произвольных nmID отдельным вызовом не собираются
   - Target: переиспользовать существующий curl-механизм (не Node fetch) для получения первых 5 фото листинга и характеристик товара по nmID; НЕ писать новый механизм скана
   - Acceptance: для тестового nmID возвращаются ≥1 (цель — 5) URL фото листинга и набор характеристик; источник — существующий механизм проекта (подтверждается ссылкой на переиспользуемую функцию в PLAN)

5. **Персистентность: несколько ниш с историей**: каждый прогон сохраняется, к нему можно вернуться.
   - Current: хранилища прогонов нет
   - Target: Prisma-модели «прогон ниши» (метаданные: дата, диапазон, статус полноты) + связанные SKU-данные (воронка, метрики, фото, характеристики, позиции); список сохранённых прогонов с возможностью открыть прошлый
   - Acceptance: после прогона запись сохранена в БД и открывается позже без повторного сбора; список показывает ≥2 разных прогона; повторный прогон создаёт НОВУЮ запись (не перезаписывает старую)

6. **Единая сортировка топ-30**: по выручке (сумма заказов) или по конверсии клик→заказ, действует на всех вкладках.
   - Current: —
   - Target: переключатель сортировки; выбранный порядок применяется одинаково во всех 5 вкладках и в PDF
   - Acceptance: смена сортировки меняет порядок строк идентично на всех вкладках; порядок в PDF совпадает с выбранным на экране

7. **Правило полноты по рангу выручки**: сбой сбора в топ-10 блокирует сохранение; в позициях 11–30 — сохраняется с пометкой.
   - Current: —
   - Target: SKU ранжируются по выручке; если по любому из **топ-10** (по выручке) не собраны обязательные данные (воронка + 5 фото + характеристики + позиции) — прогон НЕ сохраняется и показывает ошибку с указанием проблемных SKU; если проблема в рангах **11–30** — прогон сохраняется с явной пометкой «проблема с выгрузкой по SKU …»
   - Acceptance: смоделированный сбой по SKU из топ-10 → прогон не сохранён + ошибка; смоделированный сбой по SKU ранга 11–30 → прогон сохранён + видимая пометка неполноты со списком SKU

8. **Дашборд: 5 вкладок, все 30 строк видны**: во всех вкладках 1 строка = 1 SKU, видны все 30.
   - Current: —
   - Target: полноэкранный дашборд с вкладками: **Общая информация** (по умолч.: фото+бренд+артикул+продавец, рейтинг+отзывы, средняя цена/мес, сумма заказов/мес, конв. клик→заказ), **Листинг** (артикул+бренд, узкие колонки выручки и конв. клик→заказ, крупно 5 фото), **Характеристики** (артикул+бренд, выручка+конв. клик→заказ, блок характеристик), **Статистика карточки** (req. 9), **Статистика запросов** (req. 10)
   - Acceptance: все 5 вкладок открываются и показывают все 30 SKU; «Общая информация» открыта по умолчанию; каждая вкладка содержит перечисленные поля

9. **Вкладка «Статистика карточки» — графики динамики**: столбец средних + «одна метрика = один график».
   - Current: —
   - Target: слева компактный столбец средних в фикс. порядке (показы, CTR, конв. в корзину, конв. в заказ, заказы, сумма); сверху панель выбора метрик (показы, CTR, конв. в корзину, конв. в заказ, заказы, сумма, выкуп, цена — без позиций); каждая выбранная метрика = отдельный график динамики по дням в каждой строке; ось X — даты, справа шкала значений с делениями; hover — точное значение за день
   - Acceptance: выбор 1 метрики строит 1 график в каждой строке; выбор N метрик строит N графиков; ось X = даты периода; при наведении показывается значение за конкретный день

10. **Вкладка «Статистика запросов» — тепловая карта позиций**: запросы × дни.
    - Current: —
    - Target: по строке — тепловая карта позиций (запросы × дни) с подписанными датами; слева у каждого запроса цветной бейдж средней позиции за период, считается только по дням присутствия (дни отсутствия = прочерк, в среднее не входят и не штрафуют); по умолчанию видно 5 запросов, остальные — скролл внутри строки; цвет ячейки: топ — зелёный, глубоко — красный, отсутствие — прочерк
    - Acceptance: тепловая карта рендерит матрицу запрос×день; средняя позиция игнорирует дни-прочерки; по умолчанию показаны 5 запросов + доступен скролл к остальным; цветовая шкала соответствует глубине позиции

11. **PDF-выгрузка**: сводная таблица + по-SKU блоки.
    - Current: PDF-выгрузки нет (`pdfkit` доступен, но не используется для этого)
    - Target: кнопка выгрузки формирует PDF: (a) сводная таблица (артикул, бренд, суммарная выручка/мес, конв. клик→заказ = клик→корзина × корзина→заказ) + (b) по-SKU блоки: артикул+бренд, выручка/мес, конв. клик→заказ, 5 фото листинга, график цены, график конверсий (в корзину и в заказ); порядок = текущая сортировка
    - Acceptance: PDF генерируется без ошибки для сохранённого прогона; содержит сводную таблицу с 30 строками и по-SKU блок на каждый SKU с указанными полями, 5 фото и 2 графиками; порядок SKU совпадает с выбранной сортировкой

12. **RBAC-раздел ANALYTICS**: новый раздел с гейтингом доступа.
    - Current: `ERP_SECTION` не содержит ANALYTICS; `/analytics` не зарегистрирован
    - Target: добавить `ANALYTICS` в enum `ERP_SECTION` (рукописная миграция + `prisma migrate deploy`), маршрут `/analytics` в `lib/sections.ts`/`lib/rbac.ts`/навигацию; read — просмотр, write (MANAGE) — загрузка файлов / запуск MPSTATS-сбора
    - Acceptance: неавторизованный по разделу пользователь получает 403 на `/analytics`; SUPERADMIN и пользователь с грантом ANALYTICS видят раздел; запуск сбора требует MANAGE

## Boundaries

**In scope:**
- Ручная загрузка 6 detail-JSON «Сравнение карточек» и определение топ-30 из них
- Движок метрик воронки (правила ТЗ §3: среднее÷30, «от сумм», клик→заказ, цена −3%)
- Интеграция MPSTATS (позиции органика/реклама, окно = период файлов, запросы >500 из MPSTATS по SKU)
- Переиспользование существующего скана карточек WB (5 фото листинга + характеристики)
- Персистентность нескольких ниш/прогонов с историей (Prisma-модели)
- Единая сортировка топ-30 (выручка / конверсия клик→заказ) на всех вкладках
- Правило полноты по рангу выручки (топ-10 блокирует сохранение, 11–30 — пометка)
- Полноэкранный дашборд из 5 вкладок (Общая / Листинг / Характеристики / Статистика карточки / Статистика запросов)
- Графики динамики «одна метрика = один график» + тепловая карта позиций
- PDF-выгрузка (сводная таблица + по-SKU блоки)
- Новый `ERP_SECTION.ANALYTICS` + маршрут `/analytics` + RBAC

**Out of scope:**
- Автоматический сбор файлов «Сравнение карточек» — WB не даёт официального API этого отчёта; только ручная загрузка перехваченных JSON
- Привязка к товарам компании (`Product`/`MarketplaceArticle`) — раздел про КОНКУРЕНТОВ (произвольные nmID)
- Маркетплейсы кроме WB (Ozon и др.) — вне ТЗ
- Автообновление/cron прогонов — запуск вручную (прогон дорогой по лимитам MPSTATS)
- Экспорт в Excel/CSV — только PDF (по ТЗ)
- Прогнозы/рекомендации/AI-инсайты — только визуализация фактических данных
- Мульти-кабинет / несколько MPSTATS-токенов — один токен на прогон
- Модификация существующего механизма скана карточек — переиспользуем как есть, не переписываем

## Constraints

- **MPSTATS лимиты:** 1 запрос по API = 1 лимит внешней аналитики по тарифу; 30 SKU = ≥30 запросов + пагинация внутри SKU → сбор длительный, нужны прогресс и устойчивость к 429/исчерпанию лимита; не долбить агрессивным параллелизмом.
- **Скан карточек:** только через существующий curl-механизм `card.wb.ru` (Node `fetch` заблокирован TLS-fingerprint) — не изобретать новый транспорт.
- **Окно дат:** MPSTATS-период привязан к периоду `byDay` файлов (единая ось X для воронки и позиций).
- **Цена:** медианная цена отчёта отдаётся с СПП → показываем с WB-Кошельком = report_price × 0.97 (−3%).
- **Полнота:** топ-10 по выручке — блокирующее условие сохранения прогона.
- **UI:** полноэкранный (крупные графики/тепловые карты), не узкий preview-макет.
- **Стек (без новых тяжёлых зависимостей):** Next.js 15 App Router (RSC + server actions), Prisma 6/PostgreSQL, `recharts` (+shadcn chart), `pdfkit` (PDF), Auth.js v5 RBAC, sticky-table pattern из существующих разделов.

## Acceptance Criteria

- [ ] Загрузка 6 валидных detail-JSON даёт ровно 30 SKU; невалидный/дублирующий файл отклоняется с сообщением
- [ ] Движок: объёмы = месяц÷30; конверсии/выкуп «от сумм»; клик→заказ = клик→корзина × корзина→заказ; цена = report×0.97 — покрыто golden-тестами
- [ ] MPSTATS: дневные позиции по SKU разделены на organic/ad; запросы с частотностью ≤500 отфильтрованы; окно дат = период файлов; лимит-ошибка не роняет прогон
- [ ] Скан: по nmID получены 5 фото листинга + характеристики через переиспользуемый curl-механизм
- [ ] Прогон сохраняется в БД и открывается позже; список содержит ≥2 прогона; повторный прогон = новая запись
- [ ] Сортировка (выручка / конв. клик→заказ) даёт идентичный порядок во всех 5 вкладках и в PDF
- [ ] Сбой по SKU из топ-10 → прогон НЕ сохранён + ошибка; сбой в 11–30 → сохранён + пометка неполноты
- [ ] 5 вкладок открываются, показывают все 30 SKU; «Общая информация» по умолчанию; поля каждой вкладки соответствуют ТЗ §4
- [ ] «Статистика карточки»: N выбранных метрик → N графиков в строке; ось X = даты; hover = значение за день
- [ ] «Статистика запросов»: тепловая карта запрос×день; средняя позиция без учёта дней-прочерков; 5 запросов по умолч. + скролл; цветовая шкала по глубине
- [ ] PDF генерируется: сводная таблица (30 строк) + по-SKU блоки (5 фото + график цены + график конверсий); порядок = сортировка
- [ ] `/analytics` гейтится RBAC: 403 без гранта; доступ у SUPERADMIN и гранта ANALYTICS; сбор требует MANAGE
