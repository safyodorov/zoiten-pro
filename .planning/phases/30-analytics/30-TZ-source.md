# ТЗ (исходник) — Дашборд «Топ-30 SKU в нише»

> Источник: `TZ_dashboard_top30_v3.pdf`, передан пользователем 2026-07-13.
> Это дословный исходник заказчика для Phase 30 «Аналитика». SPEC.md фиксирует
> из него falsifiable-требования; здесь — оригинал без изменений смысла.

## 1. Что это и зачем

Веб-дашборд для анализа топ-30 карточек одной ниши на Wildberries. На одном
полноэкранном пространстве собрана вся картина по каждому товару — карточка,
воронка продаж, выручка, позиции в поиске — с переключением между вкладками,
сортировкой и наглядными графиками. Цель — за минуту понять, кто в топе и за счёт чего.

## 2. Входные данные

На входе два источника, которые даёт пользователь (2.1 и 2.2); листинг и характеристики
товара сервис собирает сам — сканом карточек (2.3). Всё остальное сервис считает и собирает сам.

### 2.1. Шесть файлов details (из отчёта WB «Сравнение карточек»)

Каждый файл — JSON-ответ, который отдаёт кабинет WB при открытии отчёта «Сравнение
карточек» (перехватывается через DevTools → Network → запрос `detail`). В каждом файле 5 SKU,
итого 6 × 5 = 30 SKU. **Список артикулов дашборда полностью определяется этими файлами** —
какие SKU в них, те и в топ-30; никакого дополнительного отбора сервис не делает.

Формат файла (структура, сокращённо):

```json
{
  "error": false,
  "data": {
    "ID": "<uuid сравнения>",
    "salesFunnel": {
      "byDay":  [ { "nmID", "dt", "openCard", "addToCart", "orders",
                    "ordersSum", "viewCount", "CTR", "medianPrice",
                    "buyoutCount", "buyoutPercent", "cancelCount", ... }, ... ],
      "byWeek":  [ ... те же поля, агрегировано по неделям ... ],
      "byMonth": [ ... те же поля, агрегировано по месяцам ... ]
    },
    "commonParams": [ { "nmId", "brandName", "mainPhoto", "subject",
                        "nmRating", "feedbacksCount", "feedbacksRating",
                        "orders", "viewCount", "medianPrice", "nmCreated", ... } ],
    "searchQueries": [ ... ]
  }
}
```

Ключевые поля:
- **byDay** — посуточная воронка по каждому SKU: `viewCount` (показы), `openCard` (переходы),
  `CTR`, `addToCart` (корзины), `orders` (заказы), `ordersSum` (сумма заказов, ₽),
  `buyoutPercent` / `buyoutCount` (выкуп), `medianPrice` (медианная цена).
- **commonParams** — бренд, продавец, рейтинг, число отзывов, главное фото.

Период `byDay` — примерно 30 дней, заканчивается вчерашним днём.

### 2.2. API-ключ MPSTATS

Персональный токен MPSTATS (кабинет → Аккаунт → блок «API токен»), передаётся в заголовке
`X-Mpstats-TOKEN`. Через него сервис вытягивает историю позиций карточек в поиске по запросам
(с разбивкой на органические и рекламные позиции) по каждому из 30 SKU. **Список артикулов для
MPSTATS зашит и совпадает с 30 SKU** из файлов «Сравнения карточек»; период по умолчанию —
30 дней. Запрос к API выполняется по одному SKU за раз, поэтому 30 SKU = 30 обращений к API
(плюс пагинация запросов внутри одного SKU). Про лимиты помним: **1 запрос по API = 1 лимит
внешней аналитики по тарифу.**

### 2.3. Листинг и характеристики товара (скан карточек)

Первые 5 фото листинга и характеристики товара в файлах details и в MPSTATS не приходят
(в `commonParams` есть только главное фото). Их сервис собирает **сканированием карточек товара
по тому же принципу, который уже реализован в проекте в другом функционале.** Конкретный способ
скана (источник данных, формат ответа, поля характеристик и ссылки на фото) определяем на основе
анализа проекта, к которому подключается сервис из этого ТЗ, и **переиспользуем существующий
механизм, а не пишем новый.**

## 3. Данные по каждому SKU

**Общая информация:** SKU, бренд, продавец.

**Из карточки товара:** характеристики, первые 5 фото листинга, рейтинг, количество отзывов
(источник листинга и характеристик — скан карточек, см. 2.3).

**Из «Сравнения карточек»** — в двух видах (среднее и динамика по дням):
- показы
- CTR (переходы / показы)
- конверсия клик → корзина
- конверсия корзина → заказ
- конверсия клик → заказ — расчётная: клик→корзина × корзина→заказ (в исходнике её нет)
- количество заказов
- сумма заказов (выручка)
- процент выкупа
- **медианная цена с поправкой на кошелёк:** цена в отчёте отдаётся с СПП, нам нужна цена с
  WB-Кошельком, поэтому из цены отчёта вычитаем 3%.

**Из MPSTATS:** динамика позиций в поиске по дням, отдельно органическая позиция и рекламная
позиция (с параметрами рекламы: ставка CPM, тип размещения, позиция буста). Номенклатура
запросов — **все запросы ниши с частотностью больше 500.**

### Правила расчёта

- Показы, заказы, сумма заказов — показываем как **среднее за день** (месячное значение ÷ 30).
- Конверсии и выкуп за месяц считаем **не как среднее арифметическое дневных процентов, а
  от суммарных объёмов за месяц**: сначала складываем абсолютные величины (показы, переходы,
  корзины, заказы, выкупы), затем считаем конверсии из этих сумм:
  - CTR = Σ переходов / Σ показов
  - клик → корзина = Σ корзин / Σ переходов
  - корзина → заказ = Σ заказов / Σ корзин
  - клик → заказ = Σ заказов / Σ переходов
  - выкуп = Σ выкупов / Σ заказов
- Медианная цена — **средняя за период** (уже с −3%).

## 4. Структура экрана: вкладки

Дашборд переключается вкладками; во всех вкладках видно сразу все 30 строк (одна строка = один SKU).

### Вкладка «Общая информация» (открыта по умолчанию)
По строке: фото + бренд + артикул + продавец, затем рейтинг с количеством отзывов, средняя цена
за месяц, сумма заказов за месяц, конверсия клик → заказ.

### Вкладка «Листинг»
По строке: артикул + бренд, узкие колонки выручки и конверсии клик → заказ, и крупно — первые
5 фото товара (фото приоритетны по площади, числа ужаты).

### Вкладка «Характеристики»
По строке: артикул + бренд, выручка и конверсия клик → заказ, и вместо фото — блок характеристик товара.

### Вкладка «Статистика карточки»
Слева по строке — компактный столбец средних значений в фиксированном порядке: показы, CTR,
конверсия в корзину, конверсия в заказ, заказы, сумма. Числа полные, выровнены в один столбец.
Справа — крупные графики динамики по дням, по принципу «одна метрика = один график».
Сверху таблицы — панель выбора параметров (показы, CTR, конв. в корзину, конв. в заказ, заказы,
сумма, выкуп, цена — без позиций); можно включить один или несколько, каждый выбранный строится
отдельным графиком в каждой строке. У графика по оси X — даты, справа — шкала значений с делениями
(значение за день читается без наведения), при наведении показывается точное значение за конкретный день.

### Вкладка «Статистика запросов»
По строке: артикул + бренд и тепловая карта позиций (запросы × дни) с подписанными датами.
У каждого запроса слева — средняя позиция за 30 дней (цветной бейдж), считается только по дням
присутствия в выдаче; дни отсутствия — прочерк, в среднее не входят и никак не штрафуются.
По умолчанию видно 5 запросов, остальные — прокруткой внутри строки. Цвет ячейки кодирует глубину
позиции: топ — зелёный, глубоко — красный, отсутствие — прочерк.

## 5. Сортировка

Топ-30 сортируется по одному из двух параметров на выбор: **по выручке (сумма заказов)** или **по
конверсии клик → заказ**. Сортировка действует на всех вкладках.

## 6. Масштаб отображения

Сервис полноэкранный: графики на «Статистике карточки» и тепловые карты — крупные и наглядные
(в отличие от узкого превью-макета).

## 7. Открытые вопросы (на этап логики)

- **Окно дат для MPSTATS:** период зашит по умолчанию — 30 дней. Осталось продумать точную
  привязку начала/конца окна к периоду «Сравнения карточек» и общую шкалу дат для обеих частей —
  на этапе логики.
- **Механизм скана карточек:** до реализации проанализировать проект, к которому подключается
  сервис, и понять, как в нём уже сканируются карточки (листинг и характеристики), чтобы
  переиспользовать этот механизм, а не изобретать новый.
