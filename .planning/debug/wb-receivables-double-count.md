# Investigation: двойной счёт дебиторки WB (`current + weeklyTail`)

**Дата:** 2026-07-03
**Триггер:** пользователь заметил, что баланс WB в кабинете меняется в реальном времени (35 786 842 → 36 068 177 ₽ за ~3 ч), и усомнился, что `weeklyTail` (незакрытая неделя) правомерно прибавляется к `current`.
**Вердикт:** ✅ **Подтверждён двойной счёт.** Дебиторка в Phase 24 завышена примерно на величину `weeklyTail` (на 02.07 — на ~10,37 млн ₽, ~22% строки дебиторки).

## Текущая (ошибочная) модель

`FinanceReceivablesSnapshot.totalRub = balanceCurrentRub + weeklyTailRub` (см. [lib/finance-snapshot.ts](../../lib/finance-snapshot.ts) шаг 2), где:
- `balanceCurrentRub` = поле `current` из Balance API `GET finance-api.wildberries.ru/api/v1/account/balance`
- `weeklyTailRub` = Σ `forPay` из Statistics Sales API за незакрытую неделю [понедельник, дата снапшота] ([lib/wb-finance-api.ts:107](../../lib/wb-finance-api.ts) `fetchWeeklyForPayTail`)

Допущение дизайна ([wb-finance-api.ts:103](../../lib/wb-finance-api.ts)): *«прошлые недели уже в `balance.current` → берём только текущую неделю»* — т.е. предполагалось, что `current` НЕ содержит текущую неделю.

## Доказательства

### Эмпирика (реальный аккаунт, Statistics Sales, неделя 29.06–03.07)

| Бакет по дате продажи | forPay |
|---|---|
| Прошлые недели (date < 29.06, 17 строк) | 86 467 ₽ |
| Пн–Ср (29.06–01.07) | 8 235 568 ₽ |
| Чт 02.07 | 2 136 394 ₽ |
| **Пн–Чт (29.06–02.07) = окно weeklyTail** | **10 371 962 ₽** |
| Пт 03.07 (сегодня) | 2 217 008 ₽ |

- `weeklyTail` в снапшоте 02.07 = **10 371 962,32** — совпадает копейка-в-копейку с суммой выкупов Пн–Чт ⇒ `weeklyTail` = сумма **выкупов** (Statistics Sales `forPay`) за неделю.
- Balance `current`: 35 763 515,79 (≈15:xx МСК) → 36 069 086,98 (≈19:xx МСК) = **+305 571 ₽ за ~3 ч**. За это окно измениться могли только сегодняшние выкупы (прошлых недель во всём ответе всего 86 467 ₽). ⇒ **выкупы текущей недели попадают в `current` в реальном времени.**
- `current` (36 069 087) ≈ виджет «Баланс» кабинета (36 068 178), Δ ~900 ₽.

Вывод эмпирики: `current` уже включает выкупы текущей недели; `weeklyTail` = те же выкупы ⇒ двойной счёт.

### Документация WB (независимая сверка)

- **Официальная инструкция WB «Как работать с балансом»** (seller.wildberries.ru/instructions/.../how-to-manage-balance, A-225): текущий баланс = доход **в реальном времени**, «обновляется постоянно по мере поступления транзакций»; = выручка от выкупов − комиссия − возвраты − логистика/хранение − прочие удержания. Это и есть виджет «Баланс».
- **Sales API** (`/supplier/sales`) отдаёт **выкупы и возвраты** (saleID `S…`/`R…`), `forPay` = «к перечислению» по продаже. Это те же выкупы, что формируют `current`.
- `for_withdraw` — ОТДЕЛЬНАЯ величина «доступно к выводу» (недельный/3-недельный цикл), **≠ дебиторка**.
- ⚠ `forPay` в Sales API считается по **упрощённой** логике и может отличаться от `ppvz_for_pay` в детализации отчёта реализации (`reportDetailByPeriod`) — т.е. `weeklyTail` в любом случае лишь оценка.

Оба независимых метода (эмпирика + официальная документация) дают один вывод.

## Корректная модель дебиторки WB (без двойного счёта)

Три НЕпересекающихся слоя:
1. **`current`** — уже начислено WB (все выкупы, включая текущую неделю), net от комиссии/логистики/возвратов. Из Balance API. ← это и есть «сколько WB уже должна».
2. **Σ forPay(заказы «в пути к клиенту», ещё НЕ выкупленные) × ожидаемый %выкупа** — будущая дебиторка. Из **Orders API** (`/supplier/orders`), НЕ из Sales. Sales этот слой не покрывает.
3. − резерв на невыкуп/возвраты.

**`weeklyTail` (выкупы недели из Sales API) не относится ни к одному «дополнительному» слою — он уже внутри `current`.**

### Рекомендация (v1)

**`Дебиторка = current`.** Убрать `weeklyTail` из суммы.

Обоснование против добавления слоя (2) прямо сейчас: товар «в пути к клиенту» уже стоит в балансе **как запас** (строка «WB в пути к клиенту» по себестоимости). Добавить его будущий `forPay` в дебиторку = задвоение того же актива (запас + дебиторка). Слой (2) как отдельная прогнозная строка — возможное будущее улучшение, но требует Orders API + калибровки %выкупа + аккуратной развязки с запасами.

## Требуемые правки

1. **[lib/finance-snapshot.ts](../../lib/finance-snapshot.ts):** `totalRub = current` (не `current + tail`). `weeklyTailRub` можно оставить в модели как справочное поле (аудит), но НЕ суммировать; либо перестать его тянуть (экономит 1 вызов Statistics Sales за ночь).
2. **[lib/balance-data.ts](../../lib/balance-data.ts):** строка дебиторки = `current`. Сегодняшний split (current + tail двумя строками) presented задвоенный итог — переделать: одна строка «Дебиторка WB» = `balanceCurrentRub`. `weeklyTail` — максимум как информационная под-заметка «выкупы недели уже в балансе», без включения в subtotal.
3. **Историческая правка данных:** `UPDATE "FinanceReceivablesSnapshot" SET "totalRub" = "balanceCurrentRub"` для существующих снапшотов (02.07). Дебиторка 02.07 станет ≈ 35,76 млн вместо 46,14 млн; Активы/Капитал скорректируются.
4. Тест [tests/balance-sheet.test.ts](../../tests/balance-sheet.test.ts): мок/ассерты под новую формулу.

## Открытые вопросы (не блокируют v1 = «дебиторка = current»)

- Точный лаг попадания выкупа в `current` vs `lastChangeDate` в Sales — документально не зафиксирован (эмпирически «почти realtime»).
- Если делать слой (2) «в пути × %выкупа» — калибровать `forPay`-оценку на `reportDetailByPeriod` (Sales `forPay` упрощён).
