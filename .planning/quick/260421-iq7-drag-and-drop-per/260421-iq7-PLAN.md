<objective>
В разделе «Товары» ввести drag-and-drop порядок артикулов маркетплейса (первый = основной), привязать штрих-коды к конкретному артикулу вместо товара и реализовать уникальность штрих-кода в пределах маркетплейса. Сохранить полную совместимость с WB sync и prices/wb.

Purpose:
- «Основной» артикул WB (с которого WB card sync берёт характеристики/фото) становится явным — первым в списке по sortOrder. Без этого при наличии нескольких артикулов на один товар выбор «основного» неопределён.
- Штрих-код логически принадлежит артикулу, а не товару: один и тот же GTIN может легально встречаться на WB и на Ozon, но не дважды внутри WB. Текущая схема `@unique(productId, value)` этого не гарантирует.
- UI штрих-кодов как отдельной секции отрывает их от артикула, к которому они фактически относятся, и не масштабируется при нескольких артикулах на один маркетплейс.

Output:
- Миграция Prisma применяется через `prisma migrate deploy` на проде без потери данных (2-step NOT NULL + backfill).
- Форма /products/new и /products/[id]/edit показывает reorderable список артикулов с их штрих-кодами справа.
- Server actions сохраняют sortOrder и barcodes через marketplaceArticleId.
- WB sync пишет штрих-коды с привязкой к артикулу WB.
- Post-deploy чек: `Barcode WHERE marketplaceArticleId IS NULL` → 0.
</objective>

<migration_strategy>
Файл: `prisma/migrations/20260421_article_sortorder_barcode_fk/migration.sql`

**КРИТИЧНО — ПОЧЕМУ ДЕНОРМАЛИЗАЦИЯ productDeletedAt (а не subquery в WHERE partial index):**
PostgreSQL НЕ поддерживает subquery (в т.ч. EXISTS) в predicate partial unique index. Текущий паттерн в `20260405_partial_indexes/migration.sql` (`WHERE EXISTS ... Product.deletedAt IS NULL`) — технически работает в PG (IMMUTABLE subquery разрешён для non-unique indexes), но для UNIQUE partial index PG ограничивает predicate чистыми IMMUTABLE-выражениями колонок той же таблицы. Реалистичный путь — денормализовать `Product.deletedAt` в `Barcode.productDeletedAt` и использовать predicate `WHERE "productDeletedAt" IS NULL`. Синхронизация поля делается на уровне server action (см. задачу 2.4).

Порядок шагов (атомарно в одной миграции — Postgres транзакционный DDL):

1. `ALTER TABLE "MarketplaceArticle" ADD COLUMN "sortOrder" INTEGER NOT NULL DEFAULT 0;`
2. Backfill sortOrder по createdAt внутри (productId, marketplaceId):
   ```sql
   UPDATE "MarketplaceArticle" ma
   SET "sortOrder" = sub.rn - 1
   FROM (
     SELECT id, ROW_NUMBER() OVER (PARTITION BY "productId", "marketplaceId" ORDER BY "createdAt" ASC, id ASC) AS rn
     FROM "MarketplaceArticle"
   ) sub
   WHERE ma.id = sub.id;
   ```
3. `ALTER TABLE "Barcode" ADD COLUMN "marketplaceArticleId" TEXT;`
4. `ALTER TABLE "Barcode" ADD COLUMN "marketplaceId" TEXT;`
5. `ALTER TABLE "Barcode" ADD COLUMN "productDeletedAt" TIMESTAMP(3);` — денормализация Product.deletedAt для partial unique
6. `ALTER TABLE "Barcode" ADD COLUMN "createdAt" TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP;`
7. Backfill: для каждого Barcode взять первый (ORDER BY createdAt ASC, id ASC) MarketplaceArticle его product и записать id + marketplaceId; одновременно скопировать Product.deletedAt в Barcode.productDeletedAt:
   ```sql
   UPDATE "Barcode" b
   SET "marketplaceArticleId" = first_ma.id,
       "marketplaceId" = first_ma."marketplaceId",
       "productDeletedAt" = first_ma."deletedAt"
   FROM (
     SELECT DISTINCT ON (ma."productId")
       ma.id,
       ma."productId",
       ma."marketplaceId",
       p."deletedAt"
     FROM "MarketplaceArticle" ma
     JOIN "Product" p ON p.id = ma."productId"
     ORDER BY ma."productId", ma."createdAt" ASC, ma.id ASC
   ) first_ma
   WHERE first_ma."productId" = b."productId";
   ```
8. Delete orphan barcodes (у Product нет ни одного MarketplaceArticle):
   ```sql
   DELETE FROM "Barcode" WHERE "marketplaceArticleId" IS NULL;
   ```
   **САМОЕ ОПАСНОЕ МЕСТО миграции.** Защита: deploy.sh делает pre-check ПЕРЕД `prisma migrate deploy` (см. задачу 4.1) и блокирует деплой если orphan > 0. Обход через `touch /var/deploy/skip_migrate_precheck` — только если оператор сознательно согласился потерять эти данные.
9. `ALTER TABLE "Barcode" ALTER COLUMN "marketplaceArticleId" SET NOT NULL;`
10. `ALTER TABLE "Barcode" ALTER COLUMN "marketplaceId" SET NOT NULL;`
    (productDeletedAt остаётся NULLable — null = активный товар.)
11. Добавить FK:
    ```sql
    ALTER TABLE "Barcode"
      ADD CONSTRAINT "Barcode_marketplaceArticleId_fkey"
      FOREIGN KEY ("marketplaceArticleId") REFERENCES "MarketplaceArticle"(id) ON DELETE CASCADE ON UPDATE CASCADE;
    ALTER TABLE "Barcode"
      ADD CONSTRAINT "Barcode_marketplaceId_fkey"
      FOREIGN KEY ("marketplaceId") REFERENCES "Marketplace"(id) ON DELETE RESTRICT ON UPDATE CASCADE;
    ```
12. Drop старого partial unique и FK `Barcode_productId_fkey`:
    ```sql
    DROP INDEX IF EXISTS "barcode_value_not_deleted_idx";
    ALTER TABLE "Barcode" DROP CONSTRAINT IF EXISTS "Barcode_productId_fkey";
    ALTER TABLE "Barcode" DROP COLUMN "productId";
    ```
13. Создать новый partial unique `(marketplaceId, value)` WHERE productDeletedAt IS NULL — БЕЗ subquery в predicate:
    ```sql
    CREATE UNIQUE INDEX "Barcode_marketplace_value_active_key"
      ON "Barcode" ("marketplaceId", "value")
      WHERE "productDeletedAt" IS NULL;
    ```
    Семантика: один GTIN уникален per marketplace среди активных (не soft-deleted) товаров. Тот же GTIN легально существует как Barcode(WB, 'X') и Barcode(Ozon, 'X') одновременно — это РАЗНЫЕ записи с разным marketplaceId, partial unique их не блокирует.
14. Индексы для FK lookups:
    ```sql
    CREATE INDEX "Barcode_marketplaceArticleId_idx" ON "Barcode"("marketplaceArticleId");
    CREATE INDEX "Barcode_marketplaceId_value_idx" ON "Barcode"("marketplaceId", "value");
    ```

Итог: схема и миграция синхронны, `prisma migrate deploy` применяется без dev-reset, partial unique использует только колонки самого Barcode (работает в PG).
</migration_strategy>
