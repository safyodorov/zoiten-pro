<architecture_notes>
Поток данных live-режима:
`page.tsx` (searchParams) → `loadWeeklyLiveBundle(weekStart)` → `loadWeeklyFinReportInputs(weekStart)`
→ движок `computeWeeklyFinReport` → `WeeklyFinReportOutput.articles: ArticleResult[]` → Table → модалка.

**КРИТИЧНО — почему транзит через `meta`, а НЕ через `WeeklyArticleInput`:**
Модалка получает ТОЛЬКО `article: ArticleResult` + `meta: WeeklyArticleMeta`. Движок
(`lib/finance-weekly/engine.ts:311`) собирает `ArticleResult` из скаляров и НЕ копирует лишние поля
входа. Значит, чтобы `rawQtyOrders`/`appliedBuyoutPct` дошли до модалки БЕЗ правки движка (запрещено),
их кладём в `WeeklyArticleMeta` (объявлен в `data.ts`, не в `types.ts`). `WeeklyArticleInput` и
`lib/finance-weekly/types.ts` НЕ ТРОГАЕМ — golden-тест движка (nmId 165967746) остаётся неизменным.

**Снапшот-режим:** зафиксированная неделя рендерится из `WeeklyFinReportSnapshot.payloadJson` — live
НЕ вызывается. `fixWeeklyReport` (app/actions/finance-weekly.ts:110) вызывает `loadWeeklyLiveBundle(weekStart)`
БЕЗ опций → снапшот ВСЕГДА пишется в дефолте (дисконт применён). Опция обязана быть optional-with-default.
Тумблер в снапшот-режиме СКРЫТ (payload заморожен), как скрыт редактор пулов (`!snapshot`).

**Версия снапшота:** НЕ инкрементировать `WEEKLY_SNAPSHOT_VERSION`. Добавление ОПЦИОНАЛЬНЫХ полей в
`WeeklyArticleMeta` обратно-совместимо: старые снапшоты парсятся, модалка показывает «—» для отсутствующих
полей (клавиша ветвления — `article.universe`, всегда присутствует). Инкремент версии инвалидировал бы
все снапшоты → лишняя перефиксация.

**Профиль тестов:** `data.ts` тянет Prisma → в pure-suite не тестируется (unit-suite сознательно обходит
data.ts). Ветка тумблера переиспользует уже покрытую pure `discountAppliancesByBuyout` (её НЕ трогаем).
Новой pure-логики нет → новый тест не нужен. Гейт: `tsc` + профильные vitest зелёные.
