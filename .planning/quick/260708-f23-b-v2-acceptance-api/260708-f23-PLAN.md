<objective>
Реализовать **Фазу B v2** дизайна «Плановые цены + второй фин-рез»: переключить источник
эффективных ставок логистики/хранения второго фин-реза с **v1 флэт-box** на **РЕАЛЬНЫЕ
per-склад ставки** из `/api/tariffs/v1/acceptance/coefficients` (короб), **взвешенные по нашему
стоку ОТДЕЛЬНО на бытовую технику / одежду** (срез §5, ранее отложенный). Дополнительно —
строка **«Возврат продавцу»** из `/api/v1/tariffs/return` (базовый `deliveryDumpSupReturnExpr`).

Фаза B v1 (второй фин-рез, флэт-ставки) **уже задеплоена** (quick 260707-m5v): есть
`WbBoxTariff`, `syncBoxTariffs`, `calculatePricingStandard`, 3 столбца Прибыль-std/ROI-std/Re-std,
второй блок модалки, кнопка «Тарифы складов», крон, GlobalRatesBar-ставки
(wbReturnLogisticsRub / wbLocalizationIndex). Здесь **переключаем источник ставок и уточняем
формулу** — таблица/столбцы/кнопка/крон переиспользуются.

Purpose: заменить приближённые флэт-ставки на реальные складские, дифференцированные по
направлению (одежда идёт на другие склады, чем бытовая техника → разная средняя логистика/
хранение). Плюс учесть возврат брака продавцу.

Output: `WbAcceptanceCoef` + `fetchAcceptanceCoefficients`/`fetchReturnTariffs`; pure-срез
`lib/wb-eff-coef.ts` (взвешивание по стоку per направление) → `AppSetting.wbEffCoef.appliances/
clothing`; `calculatePricingStandard` v2 (base+additional-liter, коэф уже в ставке, + возврат-
продавцу); резолвинг по направлению в page.tsx; wbReturnToSellerRub в GlobalRatesBar; модалка
показывает эфф-ставки + возврат.

**КЛЮЧЕВОЕ ОТЛИЧИЕ ставок v2 от v1:** acceptance-поля `deliveryBaseLiter`/`storageBaseLiter` —
это УЖЕ применённые per-склад ставки (коэффициент вшит: 46 × deliveryCoef 205% = 94,3
подтверждено на проде). Поэтому в формуле логистики/хранения коэффициент БОЛЬШЕ НЕ умножается
(в v1 было `×(delivCoefPct/100)`). Хранение теперь по схеме база+доп-литр, как логистика.
