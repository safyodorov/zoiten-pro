# Итог

Реализована **Фаза B v2** дизайна «Плановые цены + второй фин-рез»: источник эффективных
ставок логистики/хранения второго фин-реза («на стандартных условиях») в `/prices/wb`
переключён с **v1 флэт-box** на **реальные per-склад ставки** из
`/api/tariffs/v1/acceptance/coefficients` (короб, boxTypeID=2), **взвешенные по нашему стоку
ОТДЕЛЬНО на бытовую технику и одежду** (срез §5 дизайн-спеки, ранее отложенный в v1).
Дополнительно добавлена строка **«Возврат продавцу»** = ставка из `/api/v1/tariffs/return`
(`deliveryDumpSupReturnExpr`) × процент брака. Задеплоено на прод (7e77a8e), миграция
`20260708_wb_acceptance_coef` применена, `curl https://zoiten.pro` → 200.

## Отклонения от плана

### [Rule 1 — баг] EFF_FALLBACK v2 не наследует v1 box-дефолты (46/14/0.07) — используется v2-хардкод (94.3/28.7/0.16/0.16)

**Найдено во время:** Task 3, при написании `parseEff`/`EFF_FALLBACK` в `page.tsx`.

**Проблема:** `<what-built>` плана предписывал строить `EFF_FALLBACK` из `boxTariff.delivBase/
delivLiter/storageBasePerLiter` (v1 flat-average из `/tariffs/box`, ставки БЕЗ вшитого
коэффициента склада — например `delivBase=46`). Формула v1 умножала эти сырые ставки на
`(delivCoefPct/100)` отдельно (`46 × 205% ≈ 94.3`). Формула v2 (`calculatePricingStandard`)
**больше не делает это умножение** — она ожидает на входе уже применённые ставки (то, что даёт
`/api/tariffs/v1/acceptance/coefficients`, где `deliveryBaseLiter=94.3` уже включает коэффициент).
Если бы `EFF_FALLBACK` строился из `boxTariff.delivBase=46` (без коэффициента) и подставлялся
напрямую в v2-формулу, до первого нажатия «Тарифы складов» логистика/хранение занизились бы
почти вдвое (46 вместо ~94.3) — некорректные Прибыль-std/ROI-std/Re-std на проде сразу после
деплоя, до первой ручной синхронизации.

**Fix:** явный дополнительный запрос (уже присутствовал в тексте задания continuation-агенту,
переданном пользователем) — `EFF_FALLBACK` захардкожен на v2-типовые applied-ставки короба
(`delivBaseLiter=94.3, delivAddLiter=28.7, storageBaseLiter=0.16, storageAddLiter=0.16`, из
prod-recon в интерфейсах плана), с явным ветвлением на отсутствие/повреждённость
`AppSetting.wbEffCoef.<направление>` — а не на производную от `boxTariff`. Старый `boxTariff`/
`BOX_TARIFF_FALLBACK`/`wbBoxTariffEffective`-парсинг в `page.tsx` полностью убран (v1-код мёртв
для std-расчёта после переключения на v2; сам ключ `wbBoxTariffEffective` по-прежнему
записывается `syncBoxTariffs` и используется как v1-fallback ВНУТРИ синка при построении среза
§5 — но не читается page.tsx напрямую).

## UAT-пункты для пользователя

1. Открыть https://zoiten.pro/prices/wb — 3 столбца **Прибыль-std / ROI-std / Re-std** заполнены
   (не «—») для строк товаров с указанными габаритами. До первого нажатия «Тарифы складов» после
   деплоя — считают по v2-хардкод fallback (94.3/28.7/0.16/0.16 ₽/л).
2. Нажать **«Тарифы складов»** в шапке → toast с числом складов; в БД появляются строки
   `WbAcceptanceCoef` (короб) + `AppSetting.wbEffCoef.appliances/clothing` + обновляется
   `wbReturnToSellerRub`. После `router.refresh()` (кнопка сама вызывает `revalidatePath`)
   std-столбцы пересчитываются по реальным acceptance-ставкам, взвешенным по стоку.
3. Товары **разных направлений** (одежда vs бытовая техника, `Brand.direction.hasSizes`) при
   близких прочих параметрах должны показывать РАЗНЫЕ эфф-ставки логистики/хранения в модалке
   (справочная строка «Ставки (лог/хран, ₽/л)») — срез по стоку сработал, если распределение по
   складам действительно различается между направлениями.
4. Клик по строке → модалка → блок **«Стандартные условия»**: Логистика туда/эфф, Хранение,
   **Возврат продавцу** (новая строка), справочная строка эфф-ставок направления,
   Прибыль-std/Re-std/ROI-std; правка цены продавца пересчитывает блок realtime.
5. В **GlobalRatesBar** есть поле **«Возврат продавцу ₽»** (250 по умолчанию, уже было добавлено
   в Task 1) — правка меняет std-столбцы после `router.refresh()`.
6. Первый блок (расчёт по ИУ) и все существующие 27+3 столбца — без изменений.

## Осталось / примечания

- **Бакет «Прочие»/`needsClusterReview`** (упомянут в §5 спеки) — не выделяется отдельно в срезе,
  учитывается как обычный склад через `WbCardWarehouseStock`; искажение, если такие склады
  весомы, не диагностируется отдельно (только через общий `coveragePct`/`unmatched` warn-лог).
  Не в скоупе этого quick-тайла.
- **`AppSetting.wbBoxTariffEffective`** (v1 flat) продолжает записываться `syncBoxTariffs` и
  используется ТОЛЬКО как внутренний fallback при построении `effCoefFallback` в самом синке
  (если `WbAcceptanceCoef` окажется пуст для какого-то поля/направления) — `page.tsx` больше не
  читает его напрямую для std-расчёта (полностью заменён на `wbEffCoef.*`).
- **Localization index / return-logistics-rub** — не тронуты v2 (остались из v1, применяются
  так же: ИЛ множит логистику, `returnLogisticsRub` — амортизация обратной логистики при
  невыкупе).
