# Quick Task 260702-j52: Производство в /stock автоматически из закупок - Context

**Gathered:** 2026-07-02
**Status:** Ready for planning

<domain>
## Task Boundary

«Производство» (товар в Китае) в разделе Управление остатками должно автоматически считаться из раздела Закупки. Ручное редактирование количества Производства в /stock закрывается. При приёмке товара на склад Иваново и закрытии закупки остаток НЕ добавляется к Иваново автоматически (ivanovoStock остаётся полностью ручным — текущее поведение сохраняется, side effects не добавлять).

</domain>

<decisions>
## Implementation Decisions

### Формула количества «в Производстве» per товар
- Сумма по всем закупкам со статусом PLANNED и ACTIVE (COMPLETED исключается):
  `Σ (PurchaseItem.quantity − qty этапа WAREHOUSE этой позиции)`
- Кол-во этапа WAREHOUSE («Принят на складе») вычитается — частичная приёмка сразу уменьшает Производство, не дожидаясь закрытия закупки.
- Отрицательные значения per позиция клампить в 0 (если WAREHOUSE qty > quantity — данные кривые, не уходить в минус).

### Дата прихода на склад Иваново
- ОСТАЁТСЯ ручной (ProductIncoming.expectedDate + updateProductionArrivalDate) — не трогать.
- Закрывается только ручной ввод КОЛИЧЕСТВА (updateProductionStock + input в StockProductTable).

### Миграция существующих ручных значений
- Одноразовый пересчёт всех ProductIncoming.orderedQty из фактических закупок при деплое.
- Товары без открытых закупок → 0. Ручные значения пропадают — источник истины теперь закупки.

### UI /stock
- Вместо input — read-only число + tooltip с раскладкой по закупкам: поставщик, дата создания закупки, количество (за вычетом принятого). Ссылка на /procurement/purchases.
- Колонка «Дата прихода» остаётся редактируемым инпутом как сейчас.

### /purchase-plan — второй ручной редактор (решение добавлено после plan-check, 2026-07-02)
- Ручной ввод количества «Заказано» на /purchase-plan (ProcurementTable → upsertProductIncoming.orderedQty) ЗАКРЫВАЕТСЯ ТОЖЕ — количество там становится read-only, как в /stock.
- Дата прихода и план продаж (plannedSalesPerDay) на /purchase-plan остаются редактируемыми.
- Планирование будущего заказа = создание закупки со статусом PLANNED.

### Claude's Discretion
- Архитектура синхронизации: рекомендуется денормализация — recompute-функция `lib/production-sync.ts` (или в существующем lib), вызываемая из всех server actions закупок, мутирующих items/stages/status (create/update/delete purchase, savePurchaseItemStages). ProductIncoming.orderedQty становится machine-managed. Альтернатива (вычисление на лету в stock-data) допустима, если планировщик обоснует.
- Как передавать tooltip-данные (раскладку по закупкам) в StockProductTable — на усмотрение (JSON в data-слое /stock).
- Судьба server action updateProductionStock: удалить или оставить закрытым — на усмотрение (UI-ввод убрать обязательно).

</decisions>
