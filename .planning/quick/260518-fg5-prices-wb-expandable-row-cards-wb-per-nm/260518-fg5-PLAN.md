**Дополнительный контекст — ключевые решения архитектуры (READ THIS FIRST):**

1. **Single-product expanded state** vs Set: используем `Set<string>` (можно раскрыть несколько одновременно, как естественно — раскрыл товар A и B, чтобы сравнить). Это отличается от `/cards/wb` (single-open). Аргумент: пользователь сравнивает товары по динамике; ограничение single-open мешает.

2. **Click handler на Сводка cell**, не на photo — Сводка крупнее, естественнее для клика. Photo cell оставляем `cursor-default` без onClick. На Сводке: убираем существующий `onClick={(e) => e.stopPropagation()}` + `cursor-default`, заменяем на `onClick={handleProductClick}` + `cursor-pointer` + chevron-индикатор. **CRITICAL:** в `handleProductClick` НЕ забыть `e.stopPropagation()` — иначе всплытие сработает в `<tr onClick>` и откроется модалка.

3. **Где вставлять expanded row**: после ПОСЛЕДНЕЙ ценовой строки Product-группы. В существующем коде используется `flatMap` по cards + map по priceRows; productRowIdx инкрементируется. Простой способ: после внешнего `flatMap` для каждого product-group возвращать массив `[...priceRowTrs, optionalExpandedTr]`. Альтернатива — `<React.Fragment>` обёртка вокруг последней строки + expanded.

   **Решение:** меняем структуру так, чтобы каждый product-group возвращал свой массив элементов. В существующем коде `groups.map((group, gIdx) => {...return group.cards.flatMap(...).map(...)})` — обернём результат в массив + push expanded `<tr>` в конец. **Конкретно:**
   ```typescript
   return groups.map((group, gIdx) => {
     const rows = group.cards.flatMap((cardGroup, cardIdx) =>
       cardGroup.priceRows.map((row, rowIdx) => { return <tr>...</tr> })
     )
     const expandedRow = expandedProductIds.has(group.product.id)
       ? <ExpandedOrdersPanel ... />
       : null
     return <React.Fragment key={group.product.id}>{rows}{expandedRow}</React.Fragment>
   })
   ```

4. **Колонки colSpan для expanded row**: 4 sticky + 27 scroll = 31 колонок. Но скрытые колонки не рендерятся — нужен dynamic подсчёт. Удобнее: `colSpan = 4 + 1 + SCROLL_COLUMNS.filter(c => !hiddenColumns.has(c.key)).length` (4 sticky + 27 scroll where status не скрывается, остальное по visibility). Можно проще: `colSpan = 4 + 1 + visibleScrollCount` — но безопаснее использовать большое число (например 100) — браузеры clamp до фактического количества колонок.

   **Решение:** используем `colSpan={4 + 1 + SCROLL_COLUMNS.filter(c => !hiddenColumns.has(c.key)).length}` (precise).

5. **Sticky left для панели в expanded row**: таблица ~2480px, viewport ~1600px → панель должна оставаться видна при scroll. Внутри `<td colSpan>` используем `<div className="sticky left-0 w-[calc(100vw-100px)] max-w-full ...">`. Поскольку `<td>` уже занимает всю ширину, а sticky внутри table-cell может быть проблемным, делаем альтернативу: оборачиваем chart-content в `<div style={{ position: "sticky", left: 0, width: 'min(100%, calc(100vw - 320px))' }}>`. Для простоты — сначала пробуем `position: sticky; left: 0; width: max-content` или просто прячем panel в фиксированный block. Если sticky внутри colSpan td оказывается сложным — fallback: `<td colSpan>` без sticky, и panel будет уезжать (просто scroll-back, не катастрофично для v1). Это явно отметим в done criteria.

   **Решение для v1:** просто `<td colSpan>` без sticky — chart-content уезжает с скроллом. UX-приемлемо для первой итерации; sticky-фикс — отдельный quick task если жалоб не будет.

6. **Filter на сервере**: одна выборка `wbCardOrdersDaily.findMany` для всех visible nmId за последние 30 дней (eager), затем JS-фильтр nmId по правилу `stockQty > 0 OR (qty>0 за 30д)`. Стоимость: при ~100 продуктах × ~2 nmId × 30 дней = ~6000 строк max → дешевле lazy через server action. Уже есть индекс `@@index([nmId])` + `@@index([date])` на WbCardOrdersDaily.

7. **30-дневное окно vs 28-дневное (как в /cards/wb)**: задача говорит «за последние 30 дней» — но `fillTimeSeries` отдаёт 28-дневное окно, и `WbCardOrdersChart` рендерит ровно 28 точек. **Решение:** фильтр nmId считаем по 28-дневному окну (то, что показывает chart) — иначе nmId с продажами 29-30 дней назад покажется в expand-зоне с пустым графиком. Этот выбор согласован: фильтр и chart смотрят на одно окно.
