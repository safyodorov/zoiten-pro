## 1. Цель

4-я таблица раздела **Финансовая отчётность** (`/finance`, рядом с Баланс / ПДДС / ОПиУ-заглушка) — **понедельный фин-отчёт по WB-кабинету Зойтен**: для каждого артикула (nmId) за выбранную неделю считается юнит-экономика по факту (заказы + фактические затраты), в двух сценариях комиссии — **ИУ** и **Оферта** — с распределением пуловых затрат пропорционально выручке и сравнением план/факт. Озон и ДМ пока НЕ делаем (заложить место в модели).

Это тот же расчёт, что в `/prices/wb`, но на **фактических недельных данных** вместо плановой цены, плюс распределение общих затрат.

## 2. Расшифровка Excel-логики (что воспроизводим)

**Грейн:** `nmId × неделя`, кабинет Зойтен, split **бытовая** (direction.hasSizes=false) / **одежда** (hasSizes=true), два сценария комиссии.

**Три листа на неделю:**
- `Показатели … ВБ ИУ` / `… ВБ Оф` — ядро: юнит-экономика per nmId (различие = комиссия J + логистика N).
- `Сводная …` — роллап per артикул (ВБ ИУ / ВБ Оферта / Озон / ДМ: Прибыль/Re/Выручка) + план-факт (% выполнения плана недели/месяца) + низ листа = P&L-водопад затрат per категория.

**Расчёт per артикул (лист Показатели):**

| Поле | Формула | Смысл |
|---|---|---|
| E, F | вход | Выручка−комиссия ВБ, Кол-во заказов (из отчёта заказов) |
| G, H | E×коэфвыкупа, F×коэфвыкупа | нормировка (бытовая коэф=1; одежда — свой) |
| I | G/H | цена−комиссия / ед |
| **K** | **I/(100−J)×100** | **восстановленная цена продажи** («обратный пересчёт», point 5). J=комиссия %. В листе Оф — наоборот: K переносится из ИУ, I=K×(100−J)/100 |
| Q,R | L/H, M/H | реклама, списание за отзыв на ед |
| S,T,V,X | O×2%, K×1%, K×8%, K×2.87% | брак, джем, налог, эквайринг |
| P,U,W,Y,Z | **доля выручки × пул** | Доставка МП, Кредит, Общие, Приёмка, Хранение (см. §2.1) |
| AA | I−N−O−P−Q−R−S−T−U−V−W−X−Y−Z | прибыль / ед |
| AE, AF | K×H, AA×H | выручка, прибыль (→ Сводная) |
| AC, AD | AF/AE, AF/AB | Re продаж, ROI |

**Различие ИУ vs Оферта** (= дуал фин-рез из `/prices/wb` Фаза B): одна цена K, разная J и N.
- ИУ: J выше, N≈0 (логистика зашита в комиссию).
- Оферта: J ниже, N = полная объёмная логистика (сейчас в Excel вбивается руками; у нас = std `Л_эфф` из `calculatePricingStandard`).
Пример nmID 165967746 (4 зак, цена 11 748,8): ИУ приб +523,6 / Re 1%; Оферта −2176,6. Для тяжёлой бытовой ИУ выгоднее.

### Ключевые выводы исследования WB API

1. **Отчёт реализации закрывает 4 гэпа сразу (#3,#4,#10,#11)** + даёт выручку-после-комиссии per nmId (`ppvz_for_pay`). Эндпоинт: **`POST finance-api…/api/finance/v1/sales-reports/list` + `/detailed`** (замена умирающего `reportDetailByPeriod`, который удаляют **15.07.2026**). Токен: Персональный/Сервисный со scope «Финансы» (у нас есть `WB_FINANCE_TOKEN` — проверить тип/scope). Мы его **не вызываем**.
2. **Отзывы vs продвижение разделяются:** отзывы → `bonus_type_name`; продвижение → `deduction`. В новом `list` они слиты в `deductionSum` → нужен `/detailed` построчно с фильтром по операции.
3. **Многие удержания приходят на уровне счёта без `nm_id`** → распределение per-nmId = бизнес-правило (пропорц. выручке). **Это ровно та методология пулов, что в Excel** — ограничение API совпало с логикой отчёта. Точную рекламу лучше брать из Advert `/adv/v1/upd` (уже интегрирован), а не из `deduction`.
4. **ИРП / ИЛ через API недоступны** — только ЛК. Уже как ручной AppSetting (`wbIrpPct` 1,56%, `wbLocalizationIndex` 1,11).

## 4. Предлагаемая архитектура

### 4.1 Куда встраивается (UI/RBAC)

Новый **суб-роут `/finance/weekly`** под существующей секцией `FINANCE` (новый enum/миграция НЕ нужны):
- `app/(dashboard)/finance/weekly/page.tsx` (RSC, `requireSection("FINANCE")` + `getSectionRole`).
- `components/finance/FinanceTabs.tsx` — +1 таб `{href:"/finance/weekly", label:"Понедельный"}`.
- `components/layout/section-titles.ts` — regex `^/finance/weekly` (перед `/finance-models`).
- Таблица — новый компонент по образцу **`CashflowMatrix.tsx`** (транспонированная sticky-матрица, sticky-паттерн проекта, сплошной `bg-*`).
- Селектор недели — по образцу `PlanFactControls.tsx` (пресет «Текущая неделя» = ISO пн–вс уже реализован).

### 4.2 Модель данных (новое)

**A. `WbRealizationWeekly`** — импорт отчёта реализации, агрегат per (nmId, неделя):
`weekStart @db.Date`, `nmId`, `forPayRub`, `deliveryRub` (логистика), `storageRub`, `acceptanceRub`, `penaltyRub`, `reviewPointsRub` (баллы за отзывы), `promotionRub` (продвижение — для вычета/сверки), `deductionOtherRub`. + `@@unique([weekStart, nmId])`.
Плюс **account-level остаток** (строки без nmId) — отдельная таблица/строка nmId=0 для пулового распределения.
*Переиспользуемо за пределами отчёта — улучшит факт в Балансе/ПДДС.*

**B. `WeeklyFinPool`** (или AppSetting-ключи per неделя) — ручные/полу-авто пулы недели, которые не из WB:
`weekStart`, `deliveryToMpRub`, `overheadRub`, `creditInterestRub`, `creditOneTimeRub` (2,5%+НДФЛ). Источник: банк-выписка (если добавим опекс/капекс-флаг) или ручной ввод.

**C. `WeeklyFinReportVersion` (опц., Фаза 3)** — immutable-снапшот недели (как `SalesPlanVersion`): фиксирует входы+пулы+результат, чтобы прошлые недели не «плыли» при изменении справочников. В v1 можно считать on-the-fly + хранить только ручные пулы.

### 4.3 Движок — `lib/finance-weekly/engine.ts`

Чистая детерминированная функция `computeWeeklyFinReport(inputs)`:
- вход: массив per-nmId (orders, ordersRub, adSpend, costPrice, commIuPct, commStdPct, объём/габариты, reviewPoints, actualLogistics?, actualStorage?) + пулы недели + константы + базы распределения.
- на каждый nmId считает оба сценария (ИУ/Оферта), пул-распределение (`article_revenue / base_revenue × pool`), прибыль/Re/ROI.
- переиспользует `calculatePricing`/`calculatePricingStandard` для per-unit статей (брак/джем/налог/эквайринг/логистика-оферта/хранение-оценка) — **единый source of truth с `/prices/wb`**.
- возвращает per-nmId строки + роллап (по категории/бренду/кабинету) + водопад затрат.
Golden-тест на неделе 29.06-05.07 против Excel (ИТОГО Зойтен бытовая: выручка 12 620 348, прибыль 869 637, Re 7%).

### 4.4 План-факт

Переиспользовать `loadFactDaily` + `SalesPlanVersionDay` + `buildPlanFactReport({granularity:"week"})` и `{"month"}` → колонки «Плановая выручка», «% выполнения недели», «% выполнения месяца» (из `PlanFactBucket.deviationPct`).

### 4.5 Карта переиспользования (≈70% готово)

## 5. Фазировка

- **W1 — Импорт отчёта реализации** (`finance/v1/sales-reports/list`+`/detailed` → `WbRealizationWeekly`). Закрывает логистику-факт, хранение-факт, приёмку, штрафы, баллы-за-отзывы, выручку-после-комиссии. Крон в dispatcher. *Фундамент; полезен и Балансу/ПДДС.*
- **W2 — Движок + страница** `/finance/weekly`: `computeWeeklyFinReport`, per-nmId таблица + роллап + водопад, план-факт, дуал ИУ/Оферта. Golden-тест против Excel.
- **W3 — Пулы и снапшоты:** опекс/капекс-флаг на `BankTransaction` (или ручной ввод пулов), поля 2,5%+НДФЛ на `Loan`, недельный immutable-снапшот.

Можно v1 (W2) запустить на **оценках** (наш `calculatePricingStandard` + ручные пулы), а W1 подменить фактом позже — но пользователь явно просил числа из финотчёта, поэтому W1 приоритетен.

## 6. Что улучшаем против Excel

- Один источник расчёта с `/prices/wb` (не расходятся формулы).
- Логистика-оферта авто из std-юнитки (в Excel вбивается руками).
- Автозагрузка отчёта реализации (в Excel — ручная вставка «Удержаний»).
- Реклама per nmId из БД (в Excel — ручной разнос).
- План-факт из живого плана продаж (в Excel — статичные ссылки на «ПЛАН ОКТЯБРЬ»).
- Версионирование недели → воспроизводимость.
- **Ответ на point 5 (обратный пересчёт цены — «возможно лишний шаг»): да, для нас лишний.** В Excel вход = выручка-МИНУС-комиссия, поэтому цену восстанавливают `K=I/(100−J)×100`. У нас `WbCardFunnelDaily.ordersSumRub` — это уже ВАЛОВАЯ выручка по заказам (до комиссии), а комиссия известна отдельно (`WbCard.commFbwIu/Std`). Значит цену/выручку берём напрямую, а комиссию вычитаем — восстановление не нужно. (K пригодится только как сверка.)
