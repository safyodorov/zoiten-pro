# Методология ПДДС (/finance/cashflow)

Справка по тому, как считается план движения денежных средств (ПДДС) Zoiten ERP.
Эта же справка доступна прямо на странице ПДДС — кнопка **«Как считается»**
([components/finance/CashflowMethodologyDialog.tsx](../components/finance/CashflowMethodologyDialog.tsx)).
Источник правил — [lib/finance-cashflow/engine.ts](../lib/finance-cashflow/engine.ts) (`computeCashflow`).

## Ядро модели

**остаток(d) = остаток(d−1) + притоки(d) − оттоки(d)**

Стартовый остаток = банковские счета (₽, RUR) + касса (все кассы) на **начало первого дня горизонта** (= конец предыдущего дня). Потоки, датированные первым днём горизонта, входят в симуляцию и факт-ряд, а не в стартовый остаток — каждый день считается ровно один раз.
Консолидация: все юрлица группы («ГЕЙМ БЛОКС», «ДРИМ ЛАЙН», «ЗОЙТЕН», «ПЕЛИКАН ХЭППИ ТОЙС», «СИКРЕТ ВЭЙ», «ХОУМ ЭНД БЬЮТИ») в одной симуляции.

Горизонт: H2-2026 (01.07.2026–31.12.2026). Настраивается через `salesPlan.horizon` AppSetting.

## Притоки

### Выплаты Wildberries (основной источник)

**Формула:**
```
притоки_WB(d) = Σ buyoutsRub(d) × (wbPayoutPct / 100)
```

**wbPayoutPct** (дефолт **55%**) — доля выкупа, поступающая на расчётный счёт продавца.
55% = первое приближение: WB выплачивает ~66% (forPay) минус затраты на рекламу/ДРР (~12%).
Значение редактируемое (D-1 — v2 позволит задать per-товар из юнит-экономики).

**Тайминг выплат:**
WB выплачивает еженедельно, в **понедельник** недели выкупа +7 дней + лаг.

```
cash_date = monday_of(buyout_week) + 7 + wbPayoutLagWeeks × 7
```

**wbPayoutLagWeeks** (дефолт **1**) — число дополнительных недель задержки выплаты после базовой.
Диапазон: 0–8 недель (целое число).

**Источник данных:** плановые выкупы из активной версии плана продаж
(`SalesPlanVersionDay.planBuyoutsRub` × `pct / 100`).

## Оттоки

### Закупки (реальные)

Платежи по закупкам из `PurchasePayment` со статусом `PLANNED`.
Используется `amountRub` если задано (фактический платёж в рублях), иначе `amount × rate` (конвертация по курсу ЦБ на дату платежа).

### Закупки (виртуальные)

Предложения виртуальных закупок из плана продаж (`VirtualPurchase`).
**Антидвойной счёт:** виртуальные закупки со статусом `CONVERTED` не учитываются — они уже стали реальными и посчитаны в разделе «Закупки (реальные)».
`DISMISSED` тоже исключены (пользователь отклонил).

Тайминг: DEPOSIT — `orderDate + 3 рабочих дня`, BALANCE — `deposit + leadTimeDays`.
Соотношение 30/70 по умолчанию (из контракта `pdds-feed.ts`).

### Кредиты

Плановые выплаты тела кредита и процентов из `LoanPayment` (тип `PRINCIPAL` и `INTEREST`) за день симуляции.

### Налоги

Упрощённая модель v1:
- **7% от выкупов** — начисляется ежедневно пропорционально выкупам (налог на доходы УСН).
- **1% дополнительно в конец квартала** — квартальный итог в последний день Q3 2026 (30.09.2026) и Q4 2026 (31.12.2026).

Реклама зашита в коэффициент wbPayoutPct и в налоговую базу не входит.
v2: разделить НДС и налог на прибыль per-юрлицо.

### Операционные расходы (Опекс)

`opexMonthlyRub` (дефолт **0 ₽**) — ежемесячные операционные расходы.
Распределяются равномерно по дням месяца: `opexMonthlyRub ÷ days_in_month(d)`.

Позволяет учесть ФОТ, аренду, услуги и другие постоянные расходы не привязанные к конкретным платежам.

## Разрыв (Gap)

Если остаток на конец дня < `gapThresholdRub` (дефолт **0 ₽**), день помечается как разрыв.
`gapThresholdRub` — минимально допустимый уровень ликвидности (порог тревоги).
Нулевой дефолт = реальный кассовый разрыв (уход в минус).

В KPI-карточках: первая дата разрыва и минимальный остаток за горизонт.
На графике: красная пунктирная линия порога; строка «Остаток» в матрице подсвечивается красным при разрыве.

## Фактический ряд

Фактические остатки (банк RUR + касса) накапливаются из `BankTransaction` и `CashEntry` до сегодняшнего дня МСК. Отображаются на графике как вторая линия (подтверждённый факт vs прогноз).

## Параметры модели (допущения)

| Параметр | Ключ AppSetting | Дефолт | Диапазон |
|----------|----------------|--------|----------|
| Выплата WB % | `finance.cashflow.wbPayoutPct` | 55 | 0–100 |
| Лаг выплаты (нед.) | `finance.cashflow.wbPayoutLagWeeks` | 1 | 0–8 (целое) |
| Опекс/мес (₽) | `finance.cashflow.opexMonthlyRub` | 0 | 0–1 млрд |
| Порог тревоги (₽) | `finance.cashflow.gapThresholdRub` | 0 | 0–1 млрд |

Параметры редактируются MANAGE-пользователем через **AssumptionsBar** прямо на странице.
Изменение немедленно пересчитывает страницу через `router.refresh()`.

## Ограничения v1

- **wbPayoutPct** — первое приближение (55%). v2: per-товар из юнит-экономики (`pricing-math.ts`).
- **Налоги** упрощены: 7%+1% квартального. v2: разделение налоговых баз per-юрлицо, НДС отдельно.
- **Реклама** зашита в коэффициент payout, не выделяется отдельной статьёй оттоков.
- **Фактический баланс** — только RUR банк + касса (CNY-счета не конвертируются в v1).
- **Горизонт** жёстко задан как H2-2026; v2 — произвольный скользящий горизонт.
- **Консолидация** без межкомпанийских элиминаций (v1 всё в один котёл).
