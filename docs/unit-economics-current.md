# Юнит-экономика WB — как считается сейчас

> Описание текущего расчёта юнитки в разделе **Управление ценами → WB** (`/prices/wb`).
> Подготовлено для экономиста как основа для ТЗ по добавлению режима «обычные условия»
> (стандартная комиссия + логистика + хранение) рядом с текущим режимом «индивидуальные условия (ИУ)».
>
> **Дата:** 2026-07-06 · **Источник истины:** `lib/pricing-math.ts` (функция `calculatePricing`).
> Формулы выведены 1-в-1 из Excel «Форма управления ценами.xlsx» и сверены с рабочим примером
> (nmId 800750522).

---

## 1. Что это за расчёт

Раздел показывает по каждой карточке товара несколько ценовых строк и для каждой считает
экономику **одной проданной единицы**:

- **Текущая цена** — по факту с WB;
- **Regular / Auto акции** — по плановым ценам акций WB;
- **Расчётные (слоты 1–3)** — цены, которые задаёт пользователь.

Все строки считаются **одной и той же формулой** — различаются только цена/скидки на входе и, при
желании, точечные переопределения ставок. Модель построена **на индивидуальных условиях (ИУ)**:
комиссия берётся индивидуальная, а **логистика и хранение WB как отдельные статьи сейчас не
считаются вовсе** (ключевой момент — см. раздел 5).

---

## 2. Откуда берётся каждое число (входы)

На входе 16 параметров. Приоритет везде:
**ручное переопределение товара → источник (карточка WB / глобальная ставка / дефолт категории) → хардкод-дефолт.**

| Параметр (колонка формы) | Что это | Источник сейчас (цепочка приоритетов) |
|---|---|---|
| **Цена для установки** | цена продавца ДО скидки продавца | Текущая → `WbCard.priceBeforeDiscount` (из Prices API); Акции → восстанавливается из плановой цены акции; Расчётная → вводит пользователь |
| **Скидка продавца, %** | | `WbCard.sellerDiscount` (для акций — требуемая скидка акции) |
| **Скидка WB (СПП), %** | скидка за счёт WB | `WbCard.discountWb` — реальная СПП, тянется через `curl` к card.wb.ru |
| **WB Клуб, %** | скидка WB-клуба | `Product.clubDiscountOverridePct` → `WbCard.clubDiscount` → 0 |
| **Комиссия, %** | комиссия маркетплейса | `Product.commissionOverridePct` → **`WbCard.commFbwIu`** → `WbCard.commFbwStd` → 0 |
| **ДРР, %** | реклама | `Product.drrOverridePct` → `Subcategory.defaultDrrPct` → **10 %** |
| **Брак, %** | процент брака | `Product.defectRateOverridePct` → `Category.defaultDefectRatePct` → `AppSetting.wbDefectRatePct` → **2 %** |
| **Доставка на маркетплейс, ₽** | **фиксированные рубли** | `Product.deliveryCostRub` → **30 ₽** |
| **Закупка, ₽** | себестоимость | `ProductCost.costPrice` (из «Себестоимость партий») → 0 |
| **Процент выкупа, %** | | `Product.buyoutOverridePct` → `WbCard.buyoutPercent` → 100 — ⚠ **только отображается, в прибыли не участвует** |
| **Кошелёк, %** | WB-кошелёк | `Product.walletOverridePct` → `AppSetting.wbWalletPct` = **2.0 %** |
| **Эквайринг, %** | | → `wbAcquiringPct` = **2.7 %** |
| **Тариф Джем, %** | | → `wbJemPct` = **1.0 %** |
| **Кредит, %** | | → `wbCreditPct` = **7.0 %** |
| **Общие расходы, %** | | → `wbOverheadPct` = **6.0 %** |
| **Налог, %** | | → `wbTaxPct` = **8.0 %** |

Последние 7 ставок — глобальные (редактируются в шапке раздела), но по каждому товару их можно
переопределить. Для расчётных строк (слотов) возможен ещё один слой переопределений — per-слот.

## 3. Порядок расчёта и формулы

### Шаг А — цепочка цен (от цены продавца к цене покупателя)

```
Цена продавца          = Цена для установки × (1 − Скидка продавца %)
Цена со скидкой WB      = Цена продавца      × (1 − Скидка WB %)
Цена со скидкой клуба   = Цена со скидкой WB × (1 − WB Клуб %)
Цена с кошельком (что платит покупатель) = Цена со скидкой клуба × (1 − Кошелёк %)
```

### Шаг Б — расходы маркетплейса

⚠ **База почти всех расходов — «Цена продавца», а не цена покупателя.**

```
Эквайринг      = Цена продавца × Эквайринг %
Комиссия       = Цена продавца × Комиссия %
Реклама (ДРР)  = Цена продавца × ДРР %
Тариф Джем     = Цена продавца × Джем %
Скидка клуба ₽ = Цена со скидкой WB × WB Клуб %
```

### Шаг В — «К перечислению» (сколько WB перечислит продавцу)

```
К перечислению = Цена продавца − Скидка клуба ₽ − Эквайринг − Комиссия − Реклама − Тариф Джем
```

### Шаг Г — себестоимостные расходы

```
Брак           = Закупка × Брак %          ← база = ЗАКУПКА, не цена
Доставка       = 30 ₽ (фикс)
Кредит         = Цена продавца × Кредит %
Общие расходы  = Цена продавца × Общие расходы %
Налог          = Цена продавца × Налог %
```

### Шаг Д — прибыль и показатели

```
Прибыль    = К перечислению − Закупка − Брак − Доставка − Кредит − Общие расходы − Налог
Re продаж  = Прибыль / Цена продавца   (%)
ROI        = Прибыль / Закупка         (%)
```

## 4. Числовой пример (реальная строка, nmId 800750522)

**Вход:** Цена для установки 25 833 ₽, Скидка продавца 70 %, Скидка WB 25 %, Комиссия ИУ 32.58 %,
ДРР 10 %, Закупка 2 204 ₽, Брак 2 %, Доставка 30 ₽, Кредит 3 %, Общие 6 %, Налог 8 %, Кошелёк 2 %,
Эквайринг 2.7 %, Джем 1 %, Клуб 0 %.

| Строка | Значение |
|---|---|
| Цена продавца | 25 833 × (1 − 0.70) = **7 749.9 ₽** |
| Цена со скидкой WB | 7 749.9 × 0.75 = 5 812.43 ₽ |
| Цена с кошельком (платит покупатель) | 5 696.18 ₽ |
| Эквайринг | 7 749.9 × 2.7 % = 209.25 ₽ |
| Комиссия | 7 749.9 × 32.58 % = 2 524.92 ₽ |
| Реклама (ДРР) | 774.99 ₽ |
| Тариф Джем | 77.50 ₽ |
| **К перечислению** | 7 749.9 − 209.25 − 2 524.92 − 774.99 − 77.50 = **4 163.25 ₽** |
| Брак | 2 204 × 2 % = 44.08 ₽ |
| Доставка | 30 ₽ |
| Кредит · Общие · Налог | 232.50 ₽ · 464.99 ₽ · 619.99 ₽ |
| **Прибыль** | 4 163.25 − 2 204 − 44.08 − 30 − 232.50 − 464.99 − 619.99 = **≈ 567.68 ₽** |
| Re продаж | 567.68 / 7 749.9 = **7.3 %** |
| ROI | 567.68 / 2 204 = **25.8 %** |

## 5. ⚠ Что модель НЕ учитывает сейчас (главное для «обычных условий»)

1. **Логистика WB (доставка заказа покупателю) — не считается.** Единственная «Доставка» в формуле —
   это фиксированные **30 ₽ = наша поставка товара на склад WB** (входящая логистика FBW), а НЕ
   WB-логистика за отгрузку покупателю (за литр/объём).
2. **Хранение на складе WB — не считается** вообще.
3. **Процент выкупа не влияет на прибыль** — юнитка считается «на 1 проданную единицу», без обратной
   логистики и стоимости невыкупов/возвратов.
4. **Комиссия берётся индивидуальная (ИУ, `commFbwIu`)**, стандартная (`commFbwStd`) используется
   только если ИУ нет.
5. **База %-расходов — цена продавца**, а не цена покупателя/СПП и не объём (WB реально считает часть
   сборов иначе).
6. Считается только схема **FBW** (`commFbw*`); FBS-комиссии в БД есть, но не используются.

## 6. Что, по сути, добавляют «обычные условия» (для ТЗ)

- **Комиссия:** брать стандартную `commFbwStd` вместо ИУ (данные уже в БД из Tariffs API, обычно выше).
- **Добавить логистику WB** отдельной статьёй. Уточнить формулу: базовая ставка + за литр сверх;
  **зависит ли от объёма** (габариты Д×Ш×В у товара есть) и **платится ли за возвраты** (тогда нужен
  % выкупа).
- **Добавить хранение WB** отдельной статьёй (₽/день × объём) — уточнить, как усреднять
  (оборачиваемость / срок хранения).
- **Ввести % выкупа в формулу прибыли** — чтобы возвратная логистика и потери от невыкупа
  масштабировались корректно (сейчас параметр только справочный).

## 8. Следующий этап (юнитка → ПДДС)

Постановка поступлений в план движения денежных средств (ПДДС, `/finance/cashflow`) уже архитектурно
подготовлена: движок `computeCashflow` принимает опциональную функцию выплат
`PayoutFn(date, buyoutsRub, byProduct)`, а поток выручки уже несёт разбивку `byProduct[]`. Сейчас там
временный коэффициент 55 %, но подключить расчёт поступлений по юнитке можно будет **инъекцией
функции, не переписывая движок**.
